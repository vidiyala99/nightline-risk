# Backend deploy — Render + Neon (migrated off Railway, 2026-05-30)

The backend (FastAPI) runs on **Render** (free web service, built from
`backend/Dockerfile`); Postgres is **Neon** (free tier). Both start with **no
credit card**. Frontend stays on Vercel.

Service: **`nightline-risk-api`** → `https://nightline-risk-api.onrender.com`
Infra-as-code: `render.yaml` (Blueprint) at the repo root.

> Why a persistent server (not serverless): the app runs an in-process asyncio
> vision queue and holds in-memory state. Render web services are persistent
> processes — they just sleep when idle on the free plan.

---

## One-time setup (you run these — they need your accounts)

### 1. Neon Postgres (no credit card)
- Sign up at neon.tech → create a project (region near Render's, e.g. US East).
- Copy the **connection string** (looks like
  `postgresql://user:pass@ep-xxx.us-east-2.aws.neon.tech/neondb?sslmode=require`).
  `app/database.py` already normalizes the scheme; `sslmode=require` works with psycopg2.

### 2. Render web service (no credit card for the free plan)
- Sign up at render.com → **New → Blueprint** → connect this GitHub repo.
  Render reads `render.yaml` and provisions the `nightline-risk-api` web service.
  (Or **New → Web Service → Docker** and set Dockerfile path `backend/Dockerfile`,
  context `backend`, plan Free.)
- Set the one secret it can't generate: **`DATABASE_URL`** = the Neon string from step 1.
  (`APP_SECRET` is auto-generated by the Blueprint; the app refuses to boot without it.)
- Deploy. First build is slow (chromadb is a heavy, though unused, dep).

### 3. Verify it booted (the migration that crashed Railway must succeed here)
```bash
curl -s https://nightline-risk-api.onrender.com/api/coverage-lines    # expect a JSON array
```
First hit after idle cold-starts (~50s on free) — that's expected.

## Point the frontend + mobile at Render
- **Vercel** (frontend): set `NEXT_PUBLIC_API_URL = https://nightline-risk-api.onrender.com`
  (Production), then redeploy.
- **Mobile** (Expo): set `EXPO_PUBLIC_API_URL` the same way for builds.
- e2e specs, the e2e workflow, and `README.md` are already swapped to the Render URL.

## Seeding (optional, after first deploy)
Run seeds from your machine against Neon directly:
```bash
DATABASE_URL="<neon-connection-string>" python -m scripts.seed_demo_placements
```

## Notes / caveats
- **Free web sleeps after ~15min idle** → ~50s cold start on the next request.
  Fine for e2e (the workflow waits for warmup) and occasional demos. Upgrade the
  Render instance to a paid plan ($7/mo) for always-on if a live demo needs snappy.
- **512MB RAM** on free. The actual app fits (chromadb is installed but never
  imported — zero runtime cost). If a future eager-import OOMs, drop unused heavy
  deps (chromadb) from `requirements.txt` or upgrade the instance.
- **Neon free Postgres is permanent** (unlike Render's 90-day free Postgres) and
  needs no card — that's why we use it instead of Render Postgres.
- **Ephemeral storage:** local-disk evidence files are still lost on redeploy.
  The real fix is `STORAGE_BACKEND=s3` (`docs/go-live-readiness.md`).
- **Service name** must be unique on `onrender.com`; if Render assigns a different
  URL, tell me and I'll swap it across the repo.
- **Railway** (`backend/railway.toml`) is left for rollback; disconnect its GitHub
  auto-deploy on the Railway dashboard.
